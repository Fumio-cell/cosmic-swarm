# Cosmic-Swarm 画像ボクセル化モード 実装計画

## 0. 目的と背景
既存のCosmic-Swarm（内部名 Neural-Swarm）に「画像ボクセル化モード」を追加します。画像を読み込み、ダウンサンプリングして立体キューブ（ボクセル）の群れに変換し、Swarmの力学で漂わせる機能です。「散ったSwarm ⇄ 集まって元画像になる」という二状態を行き来できるようにします。

## 1. 現状調査の報告（フェーズ0結果）
`/Volumes/2TB/Antigravity/Neural-Swarm` のコードを調査しました。
- **描画プリミティブ**: 現在は `THREE.Points` + `BufferGeometry` + `ShaderMaterial` を使用しています（`ParticleSwarm.tsx`）。
- **更新ループと力学計算**: Swarmの位置更新（風、カオス、オーディオリアクションなど）は、JS側で `velocity` や `force` を保持する仕組みではなく、すべて**頂点シェーダー (`src/glsl/particle.vert`)** 内でのプロシージャル計算で行われています。
- **UI実装方式**: `lil-gui` は使用しておらず、`App.tsx` の `useState` を用いて独自コンポーネント（`OverlayHUD.tsx`）でスライダーUI（`<input type="range">`）を構築しています。

## 2. アーキテクチャと設計方針

「既存のswarm挙動は削除しない」「既存プロジェクトのファイル構成・流儀に合わせる」という制約に従い、以下の設計で進めます。

### コンポーネント構成（追加型設計）
既存の `ParticleSwarm.tsx` とシェーダーは変更せずそのまま残します。画像モード用の新しいコンポーネント **`ImageVoxelSwarm.tsx`** を新設し、`Scene.tsx` 内で「画像が読み込まれているか否か」でトグルレンダリングします。

### 画像解析（フェーズ1対応）
- `src/utils/imageAnalyzer.ts` を新設します。
- 指定解像度（スライダー値）で非表示のCanvasに縮小描画し、`getImageData()` で取得したRGBA配列から、アルファ値の閾値を超えるピクセルを抽出する純粋関数を実装します。

### ボクセル描画とアニメーション（フェーズ2〜3対応）
- `THREE.InstancedMesh` を1つだけ使用し、最大インスタンス数は 30,000 に制限します。
- 仕様書に沿って、JS側で状態（`curPos`, `velocity`, `homePos`）を配列で保持し、`useFrame` 内で力学計算を行って `setMatrixAt` を更新します。
- `dummy` の `THREE.Object3D` と `THREE.Color` を再利用し、ガベージコレクションの発生を防ぎます。
- 風（noise）は簡易的なSimplexNoise/CurlNoise関数をJS側に実装して計算します。

### UIの追加（フェーズ4対応）
- `App.tsx` に以下の状態を追加します：
  - `imageFile` (画像データ), `voxelResolution` (初期値64), `voxelSpacing` (初期値1.5), `windStrength` (初期値0.4), `gatherStrength` (初期値1.0)
- `OverlayHUD.tsx` に画像用のコントロールを追加し、ドロップゾーンは既存のオーディオアップロードと統合（ファイルタイプで自動判別）します。

## 決議事項（ベストな選択による確定事項）

「ベストな選択」として、パフォーマンスと既存UIとの親和性を最大化するため、以下のアーキテクチャを採用します。

1. **アニメーションの計算場所：GLSL（頂点シェーダー）**
   - **理由**: JSでの計算（1.6万〜30万ボクセル）は負荷が高く、既存の高度なプロシージャルノイズ（オーディオリアクションやカオス乱流）をJSで完全再現するのは非効率です。
   - **アプローチ**: `THREE.InstancedMesh` を使用しつつ、`onBeforeCompile`（またはカスタムの `ShaderMaterial`）を用いて、頂点シェーダー内で「`homePos`（画像配置）」と「カオスなSwarmの現在地」を `uGatherStrength` (0.0〜1.0) の値で `mix` (線形補間) させます。これにより、60fpsの超高速な描画と既存の美しいオーディオリアクティブ表現を完全に両立します。

2. **UIのドラッグ＆ドロップ：既存UIへの統合**
   - **理由**: オーディオ用と画像用で別々のドロップゾーンを設けると、画面が煩雑になります。
   - **アプローチ**: 現在の `OverlayHUD.tsx` のドロップゾーンを拡張し、読み込まれたファイルのMIMEタイプ（`audio/*` か `image/*`）を判定して自動で内部処理を分岐させます。
